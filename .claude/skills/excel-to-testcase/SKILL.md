# Excel → Test Case 변환 스킬

엑셀로 작성된 자연어 테스트 시나리오를 표준 마크다운 테스트 케이스로 정규화한다. 이렇게 하는 이유: 엑셀 셀의 자유 서술은 그대로 실행할 수 없고, 실행기(executor)가 단계별로 동작·검증·캡처하려면 **원자적 단계 + 검증 가능한 기대 결과**로 구조화되어야 하기 때문이다.

## 워크플로우

### 1. 엑셀 파싱 (결정적 — 스크립트 사용)
번들 스크립트로 엑셀을 JSON으로 추출한다. 직접 셀을 추측하지 말고 스크립트 출력을 근거로 삼는다.

```bash
node .claude/skills/excel-to-testcase/scripts/xlsx-to-md.mjs <엑셀경로> --dump-json
```

- `xlsx` 패키지가 없으면 스크립트가 안내한다: `npm i xlsx` (프로젝트 루트에서 1회).
- `--dump-json`은 시트·헤더·행을 JSON으로 출력만 한다(파일 미생성). 컬럼 구조 파악용.

### 2. 컬럼 매핑 추론
엑셀 헤더를 표준 필드로 매핑한다. 흔한 한국어/영어 헤더 매핑:

| 표준 필드 | 흔한 헤더 표현 |
|----------|---------------|
| `id` | TC, TC ID, 케이스번호, No, 번호 |
| `title` | 시나리오, 테스트명, 케이스명, Title, 항목 |
| `priority` | 우선순위, 중요도, Priority |
| `precondition` | 사전조건, 전제조건, Precondition |
| `steps` | 단계, 수행절차, 테스트절차, Steps, 재현절차 |
| `expected` | 기대결과, 예상결과, Expected, 결과 |
| `url` | URL, 대상, 경로, 화면 |

헤더가 불명확하면 첫 5행 샘플을 사용자에게 보고하고 가정한 매핑을 명시한 뒤 진행한다 (멈추지 않는다).

### 3. 단계 분해 (자연어 정규화 — 직접 수행)
`steps` 셀의 서술을 **원자적 단위 동작**으로 분해한다. 한 줄 = 한 동작. 동작 유형:
- 이동(navigate), 입력(type/fill), 클릭(click), 선택(select), 대기(wait for), 검증(assert)

예시 변환:
- 셀 원문: "로그인 후 장바구니에 상품을 담는다"
- 분해:
  1. 로그인 페이지로 이동한다
  2. 아이디에 `{{TEST_USERNAME}}` 입력
  3. 비밀번호에 `{{TEST_PASSWORD}}` 입력
  4. 로그인 버튼 클릭
  5. 상품 목록에서 첫 상품 클릭
  6. "장바구니 담기" 버튼 클릭

### 4. 기대 결과 검증 가능화
각 케이스에 **관찰 가능한** 기대 결과를 최소 1개 둔다: 텍스트 노출, URL 변경, 요소 표시/숨김, 개수 변화 등. 모호한 "정상 동작한다"는 구체화한다 (예: "장바구니 뱃지 수가 1 증가").

### 5. 자격증명 플레이스홀더화
아이디/비밀번호 등은 실제값 대신 `{{TEST_USERNAME}}`, `{{TEST_PASSWORD}}`, `{{TEST_BASE_URL}}` 로 표기한다. 실제값은 `.env`에서 executor가 주입한다.

### 6. 출력
- 통합본: `_workspace/01_testcases/testcases.md` (케이스가 적을 때)
- 분리본: `_workspace/01_testcases/TC-{번호}_{슬러그}.md` (케이스가 많거나 독립 실행이 유리할 때)
- 스키마는 `assets/testcase-template.md`를 정확히 따른다.

## 추론 표시 규칙
엑셀에 없어 추론한 모든 내용(대상 URL, 셀렉터 단서, 빠진 기대 결과)은 해당 위치에 다음 형식으로 남긴다:

```
> ⚠️ 추론: 대상 URL이 명시되지 않아 /login 으로 가정. 실행 시 확인 필요.
```

이유: executor가 런타임 스냅샷으로 실제 요소를 확인해 보완할 수 있도록, 어디가 불확실한지 명시적으로 전달해야 한다.
